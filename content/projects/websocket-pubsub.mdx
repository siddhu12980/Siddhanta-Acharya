The WebSocket Pub/Sub project demonstrates horizontal scaling of real-time
messaging using Redis as a cross-instance message broker. The architecture is
a five-service Docker Compose stack: two Express + `ws` server instances
behind an nginx reverse proxy, sharing a single Redis instance for pub/sub
communication.

## What's interesting

When a client connects, nginx round-robins the WebSocket upgrade to one of
two server instances. Each server subscribes to a shared Redis channel
(`"room1"`) using a singleton `RedisPubSubManager` that maintains dual Redis
clients — one for `PUBLISH`, one for `SUBSCRIBE` (a node-redis v4
requirement since a client in subscribe mode can't issue other commands).

When any client sends a message, its server publishes to Redis, and Redis
fans it out to all subscribers — including servers the sender isn't connected
to. This is the same pattern Socket.IO uses with its Redis adapter, but
built from raw `ws` and `redis` npm packages with no abstraction layer.

The project also implements private messaging via per-user Redis channels:
each connected user gets their own channel (keyed by UUID), so direct
messages are delivered only to the server instance where the recipient is
connected.

## What I'd do differently

The shared singleton `subClient` creates a cleanup bug — when any user
disconnects and unsubscribes from `"room1"`, it unsubscribes the entire
server instance. Production would need reference counting per channel. I'd
also replace Redis Pub/Sub with Redis Streams for message persistence, so
users who reconnect don't miss messages, and add proper authentication
instead of auto-generated UUIDs.
