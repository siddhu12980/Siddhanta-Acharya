FastChain is a from-scratch blockchain implementation in Python — no
libraries, no shortcuts. Every primitive is hand-rolled: SHA-256
proof-of-work, a Merkle tree for transaction integrity, a longest-chain
conflict resolution rule, and WebSocket-based peer sync so multiple
"miners" stay in sync through a shared FastAPI server.

## What's interesting

Proof-of-work runs on the server side, which is unusual. When a miner sends
a `{"type": "mine"}` message over WebSocket, the FastAPI server itself
performs the SHA-256 nonce search — incrementing from zero until the hash
starts with `"00000"` (five leading hex zeros, roughly one million hash
attempts on average). Once a valid block is found, the server broadcasts
`{"type": "new_block"}` to every connected miner, who can then validate
and append it to their local view of the chain.

The Merkle tree implementation is worth examining: transactions are
individually hashed, then paired and combined up the tree until a single
root hash remains. This root is stored in the block header, making it
impossible to tamper with any transaction without invalidating both the
Merkle root and the block's proof-of-work hash.

The `/hack` endpoint is deliberately left in — it corrupts a random
block's hash to let you watch `is_chain_valid()` catch the tampering in
real time. It is a better demo than any contrived example.

## What I'd do differently

Mining happens server-side with an `asyncio.Lock`, which serializes all
mining requests. In a real network, each node would mine independently and
race to broadcast first. I'd also add transaction signatures — the current
`signature: ""` placeholder means anyone can forge a transaction from any
address. And the balance computation does a full chain scan on every
validation step; a UTXO set would make that O(1) instead of O(n·m).
