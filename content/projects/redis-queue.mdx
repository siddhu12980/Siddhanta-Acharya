The Redis Job Queue started as a weekend experiment to understand how systems
like Sidekiq and Bull work under the hood — but without reaching for any
libraries. The entire queue is built on raw Redis primitives: `LPUSH` to
enqueue, `BRPOP` to dequeue, and `PUBLISH`/`SUBSCRIBE` for real-time
completion notifications.

## What's interesting

The system is a three-process architecture: an Express API server accepts job
submissions via `POST /submit` and pushes them onto a Redis list called
`"problems"`. A separate worker process runs an infinite `BRPOP` loop,
blocking until a job arrives. When it picks one up, it shells out to
`python3 -c` or `node -e` via `child_process.exec` to execute the submitted
code, stores the output in `results:{problemId}`, and publishes a completion
event on the `"problem_done"` channel.

The `LPUSH`/`BRPOP` pair gives you FIFO ordering for free, and because
`BRPOP` is atomic, you can run multiple worker instances without
double-processing — Redis guarantees each job goes to exactly one consumer.

## What I'd do differently

The system is deliberately simple — no retry logic, no dead-letter queue, no
job acknowledgment. If a worker crashes mid-execution, that job is gone. For
a learning project that's fine, but production would need at least a
visibility pattern like `RPOPLPUSH` into a processing list with heartbeat
timeouts.

The code execution via `exec()` is also a security concern — there's minimal
input escaping and no sandbox. If I rebuilt it, I'd swap `child_process` for
a proper isolate (like `isolated-vm`) and add execution timeouts. I'd also
drop Express for something lighter like Hono — the API is two routes; it
doesn't need middleware stacks.
